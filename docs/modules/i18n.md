# Internationalisation (i18n) — audit UX-13

> **Statut : fondation livrée (rc.22) + 8 zones traduites (rc.24→rc.45).** Noyau i18n léger, sans
> dépendance, langues **FR + EN**, sélecteur de langue et persistance. Le **module Fournisseurs**
> (`SupplierListView` + `SupplierDetailView`), le **module Livraisons** (`DeliveryListView`), le
> **module Paiements** (`PaymentListView`), le **module Rapports** (`SalesReportView` +
> `StockReportView`), les **listes Commandes/Produits** (`OrderListView`, `ProductListView`) et la
> **liste Catégories** (`CategoryListView`) sont
> traduits. La migration des chaînes des autres modules est
> **incrémentale** ; le français reste la source de vérité.

## Pour l'utilisateur
Un sélecteur de langue est disponible dans la barre supérieure de l'application (à côté des
notifications). Le choix (Français / English) est **mémorisé** sur l'appareil et réappliqué à la
prochaine visite. Les écrans non encore traduits restent affichés en français.

## Architecture technique
`frontend/src/i18n/index.ts` — noyau dépendance-free qui **imite l'API `t()` de vue-i18n** (les
appels restent identiques si l'on migre plus tard vers vue-i18n pour le pluriel / le formatage
nombres-dates-devises par locale).

| Élément | Rôle |
|---|---|
| `t(key, params?)` | Traduit une clé en chemin pointé (`'common.save'`) avec interpolation `{param}`. Fallback : locale active → français → la clé brute (signal visible si manquant). |
| `$t` | Même fonction, exposée **globalement** dans les templates par le plugin (`app.use(i18n)`), typée pour `vue-tsc` (`ComponentCustomProperties`). |
| `useI18n()` | Composable `{ locale, availableLocales, localeLabels, t, setLocale }` (forme vue-i18n). |
| `setLocale(l)` | Change la langue (réactif → re-render), persiste dans `localStorage` (`frynov.locale`) et met à jour `<html lang>`. |
| `LanguageSwitcher.vue` | Sélecteur compact (barre supérieure), `aria-label`. |

`locale` est un `ref` réactif : tout template lisant `$t`/`t` se re-rend au changement de langue.
La préférence de langue est non-sensible → `localStorage` est acceptable (à la différence du **token
d'auth**, qui reste en mémoire seule, cf. audit sécurité S6).

## Ajouter / traduire
- **Une clé** : l'ajouter dans `messages.fr` *et* `messages.en` (mêmes chemins). Utiliser
  `$t('section.cle')` dans le template (ou `t()` en script via `useI18n()`).
- **Une langue** : étendre `SUPPORTED_LOCALES` + `LOCALE_LABELS`, ajouter l'objet de messages.
  (Pour l'arabe : prévoir aussi `dir="rtl"` — non couvert par la fondation.)
- **Migrer une vue** : remplacer les chaînes en dur par `$t('…')` (template) / `t()` (script, p. ex.
  messages `confirm`/`alert` et fallback d'erreur). Mutualiser les libellés génériques sous
  `common.*` (name, email, phone, status, actions, notes, active/inactive, createdAt/updatedAt,
  saving, retry, genericError…) ; garder le spécifique sous le namespace du module (`suppliers.*`).
  **Modules traduits** : `NotFoundView` (pilote), **Fournisseurs** (`SupplierListView`,
  `SupplierDetailView`), **Livraisons** (`DeliveryListView`), **Paiements** (`PaymentListView`),
  **Rapports** (`SalesReportView`, `StockReportView`), la **liste Commandes** (`OrderListView`), la
  **liste Produits** (`ProductListView`) et la **liste Catégories** (`CategoryListView`). Reste : le
  gros des modules (incrémental).
  Note tests : la locale par défaut sous jsdom serait `en` → `test-setup.ts` force `fr` avant chaque
  test (les specs qui testent l'anglais appellent `setLocale('en')` puis restaurent).

## Limites (vs vue-i18n) & évolution
La fondation ne gère pas le **pluriel** par règles CLDR ni le **formatage nombres/dates/devises**
par locale. Si ces besoins apparaissent, migrer vers `vue-i18n` (mêmes call-sites `t`/`$t`) :
`npm i vue-i18n`, remplacer le contenu de `i18n/index.ts` par `createI18n`, conserver les fichiers
de messages.

## Tests
- `src/i18n/__tests__/i18n.spec.ts` — traduction par locale, interpolation, fallback clé manquante,
  persistance `localStorage` + `<html lang>`, locale non supportée ignorée, namespaces `suppliers`,
  `catalog` (statuts/colonnes + sous-namespace **catégories** : titre + confirmation interpolée),
  `orders`, `reports` ; `LanguageSwitcher` (options + bascule).
- `src/modules/suppliers/__tests__/SupplierListView.spec.ts` — re-rendu en anglais au changement
  de langue (preuve que la vue est câblée à i18n).
- Hygiène : `i18n` enregistré globalement dans `test-setup.ts` (`$t` partout, locale par défaut
  `fr`) ; toute spec qui change la locale la **restaure** (`afterEach(() => setLocale('fr'))`).
